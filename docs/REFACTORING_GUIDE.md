# ROBO Analyzer 리팩토링 및 스펙 변경 가이드

이 문서는 ROBO Analyzer 프로젝트의 대규모 리팩토링과 스펙 변경 작업을 종합적으로 설명합니다.

---

## 📋 작업 배경

### 왜 리팩토링이 필요했나?

1. **명칭 혼란**: "UNDERSTANDING"이라는 용어가 "ROBO-ANALYZER"와 맞지 않음
2. **복잡한 메서드 호출 처리**: 별도 프롬프트, 임시 노드 생성 등 복잡한 로직
3. **성능 문제**: 순차 처리로 인한 느린 분석 속도
4. **예외 처리 미흡**: 조용히 실패하는 부분이 많아 디버깅 어려움

### 핵심 설계 원칙

**"모든 파일에 대해 AST를 먼저 뽑는다"**

이것이 모든 설계의 핵심입니다:

```
┌─────────────────────────────────────────────────────────┐
│  모든 파일 → AST 구조 먼저 생성 (Phase 1)                │
│  ↓                                                       │
│  모든 파일 → LLM 분석 (Phase 2)                         │
└─────────────────────────────────────────────────────────┘
```

**왜 이렇게 해야 하나?**

1. **메서드 호출 처리 단순화**
   - AST 구조가 있으면 `target.methodName`을 파싱해서 기존 클래스/메서드 노드를 바로 찾을 수 있음
   - 임시 노드(TEMP) 생성 불필요
   - 복잡한 패턴 매칭 불필요

2. **병렬 처리 효율성**
   - 모든 파일의 AST가 먼저 있으면, LLM 분석 단계에서 각 파일을 독립적으로 병렬 처리 가능
   - 파일 간 의존성 없이 병렬 실행 가능

3. **관계 연결 단순화**
   - `MATCH`로 기존 노드를 바로 조회
   - `MERGE` 대신 `MATCH` 사용 가능 (AST가 이미 생성되어 있으므로)

### 2. 2단계 분석 아키텍처

#### Phase 1: AST 그래프 생성 (모든 파일)

```
모든 파일을 병렬로 처리 (5개 동시):
  ┌─ File1 ─┐   ┌─ File2 ─┐   ┌─ File3 ─┐
  │ AST     │   │ AST     │   │ AST     │
  │ 노드생성│   │ 노드생성│   │ 노드생성│
  └─────────┘   └─────────┘   └─────────┘
```

**생성되는 것**:
- CLASS, INTERFACE, ENUM 노드
- METHOD, FIELD, PARAMETER 노드
- PARENT_OF, NEXT 관계 (코드 구조)

**Cypher 쿼리 패턴**:
```cypher
-- 클래스 노드 생성
MERGE (n:CLASS {class_name: 'OrderService', user_id: '...', project_name: '...'})
SET n.startLine = 10, n.directory = 'com/example', n.file_name = 'OrderService.java', ...
RETURN n

-- 일반 노드 생성
MERGE (n:METHOD {startLine: 20, directory: '...', file_name: '...', user_id: '...', project_name: '...'})
SET n.name = 'save', n.endLine = 35, ...
RETURN n
```

#### Phase 2: LLM 분석 (모든 파일)

```
모든 파일을 병렬로 처리 (5개 동시):
  ┌─ File1 ─┐   ┌─ File2 ─┐   ┌─ File3 ─┐
  │ LLM     │   │ LLM     │   │ LLM     │
  │ 분석    │   │ 분석    │   │ 분석    │
  └─────────┘   └─────────┘   └─────────┘
```

**생성되는 것**:
- summary (요약)
- CALLS 관계 (메서드 호출)
- DEPENDENCY 관계 (타입 의존성)
- ASSOCIATION, COMPOSITION 관계 (클래스 관계)

**Cypher 쿼리 패턴**:
```cypher
-- 요약 업데이트 (MATCH 사용 - AST가 이미 있으므로)
MATCH (n:METHOD {startLine: 20, directory: '...', file_name: '...', ...})
SET n.summary = '주문을 저장합니다...'
RETURN n

-- CALLS 관계 (MATCH로 기존 클래스 조회)
MATCH (src:CLASS {startLine: 10, ...})
MATCH (dst) WHERE dst:CLASS AND toLower(dst.class_name) = toLower('OrderRepository')
  AND dst.user_id = '...' AND dst.project_name = '...'
MERGE (src)-[r:CALLS {method: 'save'}]->(dst)
RETURN r
```

#### Phase 3: User Story 생성

- 모든 분석 완료 후
- Neo4j에서 요약 데이터 조회
- User Story 문서 생성

### 3. 메서드 호출 처리 단순화

#### 이전 방식 (복잡함)

```python
# 1. 별도 프롬프트로 메서드 호출 추출
method_call_result = await invoke_method_call_analysis(...)

# 2. 임시 노드 생성
MERGE (temp:TEMP {class_name: 'OrderRepository'})

# 3. 나중에 실제 노드로 교체
MATCH (temp:TEMP {class_name: 'OrderRepository'})
MATCH (real:CLASS {class_name: 'OrderRepository'})
DELETE temp
MERGE (src)-[:CALLS]->(real)
```

#### 현재 방식 (단순함)

```python
# 1. 일반 분석 프롬프트에 calls 배열 통합
{
  "analysis": [{
    "calls": ["orderRepo.save", "validator.check"]
  }]
}

# 2. 파싱
target, method = call.split(".", 1)  # "orderRepo.save" → ("orderRepo", "save")

# 3. AST에서 직접 조회 (Phase 1에서 이미 생성됨)
MATCH (src:CLASS {...})
MATCH (dst:CLASS) WHERE toLower(dst.class_name) = toLower('orderRepo')
MERGE (src)-[:CALLS {method: 'save'}]->(dst)
```

**프롬프트 통합**:
- `rules/framework/method_call.yaml` 삭제
- `rules/framework/analysis.yaml`에 `calls` 배열 추가

### 5. 예외 처리 엄격화

#### 이전 (부분 실패 허용)

```python
try:
    result = process()
except Exception:
    return []  # 조용히 실패
```

#### 현재 (즉시 중단)

```python
try:
    result = process()
except Exception as e:
    logging.error("[ANALYZE] 처리 실패 | error=%s", e, exc_info=True)
    raise AnalysisError(f"처리 실패: {e}") from e
```

**예외 처리 정책**:
- 빈 데이터 → 예외 발생 (토큰 낭비 방지)
- LLM 응답 오류 → 예외 발생
- 파일 읽기 실패 → 예외 발생
- User Story 생성 실패 → 예외 발생

**예외 계층**:
```
RoboAnalyzerError (기본)
├── ConfigError (설정 오류)
├── AnalysisError (분석 오류)
├── CodeProcessError (코드 처리 오류)
├── LLMCallError (LLM 호출 오류)
├── QueryExecutionError (Neo4j 쿼리 오류)
└── AuthenticationError (인증 오류)
```

## 📊 결과물 형태 (변경 없음)

### 노드 속성

#### Framework (Java/Kotlin)
- `startLine`, `endLine`: 코드 위치
- `name`: 이름
- `class_name`: 클래스명 (CLASS/INTERFACE/ENUM)
- `summary`: AI가 생성한 요약
- `node_code`: 원본 코드
- `user_id`, `project_name`: 프로젝트 식별자

#### DBMS (PL/SQL)
- `procedure_name`: 프로시저 이름
- `schema`: 스키마명
- `summary`: AI가 생성한 요약
- `startLine`, `endLine`: 코드 위치

### 관계 타입

#### Framework
- `PARENT_OF`: 부모-자식 (클래스 → 메서드)
- `NEXT`: 형제 순서 (메서드1 → 메서드2)
- `EXTENDS`, `IMPLEMENTS`: 상속/구현
- `CALLS`: 메서드 호출
- `DEPENDENCY`: 타입 의존성
- `ASSOCIATION`, `COMPOSITION`: 클래스 관계

#### DBMS
- `CALL`: 프로시저 호출
- `FROM`: 테이블 읽기
- `WRITES`: 테이블 쓰기
- `HAS_COLUMN`: 테이블 → 컬럼
- `FK_TO_TABLE`: 외래키 관계

## 🔧 기술적 세부사항

### MERGE vs MATCH 사용 기준

**MERGE가 필요한 경우**:
- Phase 1 노드 생성 (중복 방지)
- DDL 테이블/컬럼 (여러 DDL에서 참조 가능)
- 관계 생성 (동일 관계 중복 방지)

**MATCH 사용**:
- Phase 2 요약 업데이트 (AST가 이미 있으므로)
- CALLS 관계 생성 (기존 클래스 조회)

**패턴**:
```cypher
-- 노드는 MATCH로 조회
MATCH (src:CLASS {...})
MATCH (dst:CLASS {...})

-- 관계만 MERGE
MERGE (src)-[r:CALLS]->(dst)
```

#### 결과 메시지화 (format_graph_result)

**이전**: Neo4j 결과를 그대로 전송
**현재**: `format_graph_result()` 함수로 사용자 친화적 메시지 생성

```python
# Neo4j 결과
graph = {
    "Nodes": [
        {"Labels": ["CLASS"], ...},
        {"Labels": ["METHOD"], ...},
        {"Labels": ["METHOD"], ...}
    ],
    "Relationships": [
        {"Type": "CALLS", ...},
        {"Type": "CALLS", ...}
    ]
}

# format_graph_result() 적용
graph_msg = format_graph_result(graph)
# 출력:
#   → CLASS 노드 1개 생성
#   → METHOD 노드 2개 생성
#   → CALLS 관계 2개 연결

# emit_message()로 전송
yield emit_message(f"      {line}")  # 각 줄을 메시지로 전송
```

**실제 사용 예시**:
```python
# framework_analyzer.py
graph = await client.run_graph_query(queries)
graph_msg = format_graph_result(graph)

yield emit_message(f"   ✓ [{completed}/{total}] {result['file']}")
if graph_msg:
    for line in graph_msg.split("\n")[:3]:  # 최대 3줄
        yield emit_message(f"      {line}")
```

**DBMS 분석 예시**:
```python
# dbms_analyzer.py
graph = await client.run_graph_query(queries)
graph_msg = format_graph_result(graph)

if graph_msg:
    yield emit_message(f"      [{analyzed_blocks}/{total_llm_batches}] 분석 완료")
    for line in graph_msg.split("\n"):
        yield emit_message(f"      {line}")
```

**"모든 파일에 대해 AST를 먼저 뽑는다"**

이것이 모든 설계의 핵심입니다:

1. **단순성**: 메서드 호출 처리가 단순해짐
2. **효율성**: 병렬 처리가 효율적
3. **정확성**: AST 기반 직접 조회로 정확도 향상
4. **유지보수성**: 임시 노드 제거로 구조 단순화
